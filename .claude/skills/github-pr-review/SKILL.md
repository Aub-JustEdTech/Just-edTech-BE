# Review GitHub Pull Request

Follow every step in order.

## Step 1 — Identify the PR

If the user provided a PR number or URL, use it. Otherwise:

```bash
gh pr list
```

Ask the user which PR to review before continuing.

## Step 2 — Fetch PR metadata

```bash
gh pr view <number> --json number,title,headRefName,author,body
```

Display: title, branch name, author.

## Step 3 — Check PR body compliance

The PR body must contain all four sections. Flag any that are missing as **blocking**:

- `### Ticket` — must have an issue or ticket reference
- `### Summary` — must be non-empty
- `### Affected Areas` — must list at least one area
- `### Tests` — must state test status

## Step 4 — Check branch name compliance

The `headRefName` must match `{type}_{description-in-kebab-case}`.

Valid types: `feat`, `fix`, `docs`, `style`, `refactor`, `test`, `perf`, `ci`, `build`, `revert`, `chore`, `hotfix`, `bugfix`, `epic`

- Wrong separator (e.g. `feature/` or `feat-`) → blocking finding
- Type not in the valid list → blocking finding

## Step 5 — Fetch the diff

```bash
gh pr diff <number>
```

## Step 6 — Check commit messages

```bash
gh pr view <number> --json commits --jq '.commits[].messageHeadline'
```

Every commit must follow Conventional Commits: `type[(scope)]: description`
- Example: `feat(heatmap): add county citations paginated endpoint`
- Flag any commit that doesn't match as a blocking finding

## Step 7 — Review diff against project rules

Check the diff for violations of the rules in `.claude/rules/`:

**Backend core rules** (`backend-core-rules.md`):
- FastAPI layered architecture respected: HTTP → Router → Endpoint → Service → CRUD → Model
- No business logic in endpoints or models — belongs in `app/services/`
- All endpoints use an appropriate auth dependency — `get_current_user`, `get_current_tenant_admin`, `get_chat_consumer_from_uuid`, `get_principal_with_api_key`, etc. — never inline token verification

**API rules** (`api-rules.md`):
- All responses use `success_response()` or `error_response()` from `app/utils/response.py`
- No raw `dict` returns from endpoints
- Query/path params validated with `Query(...)` or `Path(...)` constraints

**Database rules** (`database-rules.md`):
- Alembic migrations include a working `downgrade()` implementation
- No raw SQL strings — use SQLAlchemy ORM
- New models inherit from `BaseModel`
- New migrations are autogenerated (`alembic revision --autogenerate`) and reviewed before applying

**Code quality rules** (`code-quality-rules.md`):
- No broad `except Exception` without re-raise or structured logging
- Celery tasks use `bind=True`, `max_retries=3`, exponential backoff
- Type hints on all function signatures

**Error handling rules** (`error-handling.md`):
- Services raise `NotFoundError` / `ValidationError` / `UnauthorizedError` — not raw `HTTPException`
- CRUD layer returns `None` for not-found, does not raise domain exceptions
- `register_exception_handlers(app)` wired in `main.py`

**Pydantic schema rules** (`pydantic-schemas.md`):
- `@field_validator` used (Pydantic v2), not `@validator`
- `model_config = ConfigDict(from_attributes=True)` present on response schemas
- Separate request / response schemas — no schema doubles as both

**Type safety rules** (`type-safety.md`):
- Return type annotated on every function (`-> ReturnType`)
- No bare `Any` unless justified with a comment
- `| None` for optional values — not `Optional[X]` unless targeting < Python 3.10

## Step 8 — Produce review report

```
## PR Review: #<number> — <title>
**Branch:** <headRefName>  |  **Author:** <author>

### Compliance
- Branch name (`{type}_{description}`): ✅ / ❌ <reason if failing>
- Commit message format (Conventional Commits): ✅ / ❌ <offending commits>
- PR body complete: ✅ / ❌ <missing sections>

### Blocking Findings
(Must be resolved before merge)
- `<file>:<line>` — <description>

### Advisory Findings
(Nice to fix, not blocking)
- `<file>:<line>` — <description>

### Verdict
✅ Looks good — ready for merge  /  ❌ Changes requested
```

If there are no findings in a section, write "None."

## Step 9 — Stop

Do **NOT** run `gh pr review --approve` or merge the PR unless the user explicitly asks after seeing the report.
